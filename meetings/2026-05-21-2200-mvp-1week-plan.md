### 개발

**핵심 주장**: LNbits self-hosted + R2 presigned URL 업로드 조합으로 7일 가능. MVP에서 트랜스코딩 완전 제외.

**DB 모델**:
| 테이블 | 핵심 컬럼 |
|--------|----------|
| `users` | id, username, email, password_hash, lightning_address, created_at |
| `videos` | id, user_id, r2_key, cdn_url, file_hash, duration_sec, status, created_at |
| `posts` | id, video_id, user_id, caption, like_count, view_count, created_at |
| `reward_points` | id, user_id, week_label, points, reason, created_at |
| `lightning_claims` | id, user_id, week_label, points_used, satoshi_amount, ln_address, status, payment_hash, created_at |

**핵심 API**:
```
POST /api/v1/auth/register
POST /api/v1/auth/login
POST /api/v1/videos/presigned-url
POST /api/v1/videos/confirm
GET  /api/v1/feed
GET  /api/v1/rewards/summary
POST /api/v1/rewards/claim
GET  /health
```

**영상 업로드 플로우**:
```
Client → POST /presigned-url → Server → {upload_url, r2_key}
Client → PUT upload_url (R2 직접 업로드, 백엔드 무통과)
Client → POST /videos/confirm → DB 저장, 포인트 적립
```

**Lightning 권장**: LNbits self-hosted (Railway 별도 서비스). Strike는 한국 지역 제한 가능성. Day 3~4에 병행 셋업 필수.

**Day별 개발 일정**:
| Day | Frontend | Backend | 인프라 |
|-----|----------|---------|--------|
| 1 | 프로젝트 셋업, 라우터 | FastAPI 셋업, DB 모델, Alembic, Auth API | Railway 생성, R2 버킷, env |
| 2 | 로그인/회원가입 UI | Feed API, Like/View API | R2 CORS, CDN 도메인 |
| 3 | 피드 컴포넌트 (스와이프·자동재생) | Presigned URL API, confirm API | R2 presigned 권한 설정 |
| 4 | 영상 업로드 화면 | 포인트 적립 로직, 집계 API | LNbits 셋업 시작 |
| 5 | 리워드·claim 화면, 지갑 주소 입력 | Lightning claim API (LNbits 연동) | LNbits 완성, 테스트 지급 |
| 6 | 마이페이지, 운영자 대시보드 | 관리자 API, rate limiting | PostgreSQL 전환, env 점검 |
| 7 | 버그 수정, 모바일 최적화 | 통합 테스트 | Dockerfile 최종화, 배포 검증 |

**개발 핵심 의견**: Day 5 Lightning 연동이 critical path. 100포인트=1,000 sats 하드코딩으로 시작, 최소 claim 단위 1,000 sats 이상 설정 필수.

### QA

**검증 기준**:
- 영상 업로드 성공률 > 95%
- 피드 로딩 < 3s (모바일 4G)
- Lightning 지급 성공률 > 90%
- 포인트 적립 누락률 < 1%

**실패 시나리오 TOP 3**:
1. **Lightning invoice 만료** → 즉시 처리(동기) + 재claim 유도 UI 필수
2. **R2 업로드 성공 후 DB 포인트 누락** → idempotency key + R2 업로드 성공 후에만 DB commit
3. **동일 영상 반복 업로드 어뷰징** → 파일 해시 중복 감지 + 일일 업로드 상한(5회) + 영상 최소 10초

**운영 리스크**:
- 법적: BTC 정기 지급 → VASP 분류 가능성 검토 필요. 약관에 "본인 촬영 운동 영상만" 명시
- 보안: LNbits API 키 Railway 환경변수 엄격 관리. JWT + rate limiting 필수
- 데이터: SQLite → PostgreSQL 마이그레이션 누락 시 재배포마다 초기화

### 악마의 변호인

**① Lightning 지갑 보유율 문제**: 한국 일반 사용자 Lightning 지갑 실사용자 1% 미만. 지갑 없으면 claim 불가 → 포인트 쌓인 사용자의 미수령 BTC를 운영자가 보관하게 되면 사실상 수탁 구조로 전환됨.

**② 7일 개발 범위 과소평가**: 영상 업로드→R2→피드→포인트→Lightning 파이프라인은 최소 15~20개 API. 하나의 버그가 전체 파이프라인을 멈춤. 실제 1인 기준 3~4주 분량.

**③ 광고 수익 100만원 불가능**: MAU 100명, 1인 30PV, RPM $1 → 월 2,100원. 100만원은 MAU 1,400명 이상 필요.

**④ Railway 한계**: Hobby 플랜 RAM 512MB → 동시 영상 업로드 수 건에 OOM 가능. 100명 동시 피드 로딩 시 컨테이너 다운 리스크.

**⑤ VASP 규제**: 비수탁이라도 "운동 → BTC 지급" 플랫폼은 가상자산사업자 분류 가능성. 런칭 후 규제 해석에 따라 서비스 중단 리스크.

**⑥ Lightning 노드 유동성**: 채널 강제 종료 시 온체인 수수료가 지급액 초과 가능. 24/7 운영 부담.

## R2 토론

### 기획 (R2)
개발의 "LNbits Day 5 구현" 제안을 수용하되, planner 원안대로 **운영자 수동 정산을 병행 유지**한다. LNbits 자동화가 실패해도 수동 정산으로 첫 주 지급이 가능한 이중 구조가 필요하다. 악마의 변호인이 제기한 Lightning 지갑 보유율 문제는 실제 런칭 후 데이터로 확인해야 하므로, 온보딩에 "Lightning 지갑 설치 가이드" 링크를 필수로 포함시키는 것으로 마케팅과 합의.

### 개발 (R2)
재무가 제기한 Cloudflare Stream 비용 문제 수용 → **Cloudflare Stream 제외, R2 직접 서빙**으로 확정. 디자인이 제안한 "Lightning 주소 프로필 사전 등록" → DB 설계에 `users.lightning_address` 컬럼 이미 포함되어 있어 즉시 반영 가능. 악마의 변호인 Railway RAM 우려 → **Railway Pro 플랜($20/월, RAM 2GB) 또는 업로드는 R2 직접 업로드로 백엔드 부하 완전 제거**로 대응.

### 디자인 (R2)
마케팅이 강조한 "첫 claim 성공 영상"의 중요성 동의 → 리워드 화면에 **"BTC 받기 성공" 애니메이션과 공유 버튼**을 MVP에 포함시킨다. QA가 제기한 Lightning 주소 입력 이탈 리스크 대응 → claim 화면에 "지갑 없나요? → 추천 지갑 링크" 카드 추가.

### QA (R2)
기획의 "수동 정산 병행" 동의 → 자동 지급 실패 시 수동 처리 경로가 명확하므로 실패 시나리오 1번(invoice 만료) 심각도 낮아짐. 악마의 변호인이 제기한 어뷰징 문제 → **Day 1에 파일 해시 중복 검사 + 일일 5회 상한 구현**을 개발 일정에 명시 요청.

### 재무 (R2)
마케팅의 "첫 50명 2배 포인트" 제안 → 재무 관점에서 수용 가능. BTC 지급 재원 초기에는 광고 수익 없으므로 **초기 운영 예비비(3개월 분 인프라 + 리워드 재원, 최대 50만원) 확보 권장**. 악마의 변호인이 제기한 최소 지급 단위 → **1,000 sats 최소 claim 임계값** 코드에 하드코딩.

### 마케팅 (R2)
기획의 "7일 런칭 가능성" 동의. 악마의 변호인이 제기한 VASP 우려 → 약관에 "서비스 내 포인트는 법정화폐가 아니며, Lightning 지급은 사용자 요청에 의한 비수탁 전송" 문구 명시로 리스크 최소화(법률 검토 병행). "한입 비트코인" 브랜드 채널 없이 이 일정은 불가능하다는 점을 전체 팀이 인지해야 한다.

### 악마의 변호인 (R2)
**수용**: 개발의 R2 직접 업로드 → 백엔드 부하 제거로 Railway 한계 우려 일부 해소. 재무의 최소 1,000 sats 임계값 → 수수료 초과 문제 해소.

**유지**:
- Lightning 지갑 보유율 1% 문제는 미해소. "가이드 링크 제공"으로 해결되지 않는다. 온보딩에서 지갑 설치까지 완료하는 비율을 측정하고, 낮으면 **Wallet of Satoshi / Strike 같은 custodial 지갑 허용 여부를 v1.1에서 재논의** 필요.
- 7일 일정은 Lightning Day 5 연동이 실패하면 전체 붕괴. **Day 3에 LNbits 연결 테스트 완료 체크포인트** 없으면 경고.

**신규 경고**: "한입 비트코인" 채널에 의존한 초기 유입 전략 → 이 채널의 팔로워가 운동 습관을 보유한 비율과 Lightning 지갑 설치 의향 비율을 런칭 전 사전 설문으로 확인하지 않으면 마케팅 전환율이 예측 불가.

## 합의 및 결정

### 합의된 사항

1. **결정** — MVP 범위: 영상 업로드(R2 직접 업로드) + 세로 피드 + 포인트 적립(일일 5회·해시 중복 제한) + 주간 claim + Lightning 지급 (LNbits 자동 + 수동 정산 병행). Cloudflare Stream 제외.

2. **결정** — 기술 스택 확정: React + Vite + TailwindCSS + shadcn/ui / FastAPI + SQLAlchemy + Alembic / SQLite(개발)→PostgreSQL(프로덕션) / Cloudflare R2 / LNbits on Railway / Railway 배포.

3. **결정** — Lightning 최소 claim 단위: **1,000 sats** (수수료 초과 방지). 100포인트 = 1,000 sats 환율 하드코딩 시작.
