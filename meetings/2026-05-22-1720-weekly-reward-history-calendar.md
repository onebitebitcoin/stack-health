## R1 의견

### 기획 (R1)
- 안건 1: Phase A 포함. `GET /admin/weekly-summary` (현재 주 유저별 포인트 합계 + claim 현황) 신규 API 추가 타당.
- 안건 2: Phase B 이월. MVP 출시 D-6 상황에서 추가 불가. 런칭 후 1주일 이내 데이터 수집 후 캘린더 설계 정확도를 높이는 것이 맞음.
- 감독자 결정 필요: 리워드 표시 단위(sats vs 원화 환산), 스코어 계산 기준.

### 개발 (R1)
- `lightning_claims(user_id, week_label)` UniqueConstraint 누락 = P0 이중 지급 버그.
- 안건 1 API: `GET /admin/weekly-summary` — `SELECT user_id, sum(points) FROM reward_points WHERE week_label = :week GROUP BY user_id ORDER BY sum(points) DESC`
- 안건 2: 기존 `/api/v1/rewards/claims` 재활용으로 Profile 탭 미니뷰 구현 가능. 공수 0.5일.
- 예상 공수 안건 1: 0.5일, 안건 2 Phase B: 1.5일.

### 디자인 (R1)
- 초기 제안한 4번째 탭 추가는 철회. 탭 4개는 모바일에서 아이콘+텍스트 잘림 발생.
- **수정안**: 피드 페이지 상단에 Segmented Control (이번 주 / 지난 주) — 주간 리더보드 토글.
- 안건 2: Profile 페이지 내 "운동 기록 보기" flat CTA 버튼 → `/history` 미래 연결 준비.

### QA (R1)
- `(user_id, week_label)` 중복 방지: 애플리케이션 레이어만으로는 race condition 존재. DB UniqueConstraint P0.
- 필수 테스트 케이스: KST 주 경계(일 23:59→월 00:00), 지난 주 포인트 격리, ban 사용자 제외.
- 안건 2 Phase B 결정 시 `GET /api/v1/rewards/claims` 페이지네이션 부재 리스크.

### 악마의 변호인 (R1)
- "Blink 실계정 없으면 리더보드는 숫자 놀음" — 지급 없는 순위는 사용자 신뢰를 오히려 깎을 수 있다.
- DB 직접 조회로 대체 가능하다는 주장: admin UI가 없어도 운영 가능하다.
- 리더보드 바이럴 효과는 미검증 가정.

### 재무 (R1)
- 초기 상위 10명 집중 지급 vs 100명 분산 지급 시뮬레이션: 상위 10명 집중이 "큰 당첨금" 마케팅 효과.
- Phase 0 (런칭 전): Blink 연결 + 테스트. Phase 1 (런칭 후 4주): 운영비 예산. Phase 2 (4주+): 지급 규모 확대.
- Blink 수수료 구조 미파악 — 비용 계산 불가.

### 마케팅 (R1)
- "Real Bitcoin" 메시지 차별화. 경쟁앱(Sweatcoin) 포인트/코인과의 명확한 구분.
- Blink 실계정 연결 전까지 "첫 지급 예정" 대기 콘텐츠 준비.
- 리더보드 공개 → 1위 유저 인터뷰 콘텐츠 → 소셜 공유 유도.

### 리서처 (R1)
- Sweatcoin PMC 논문(n=5,892): 저단가 리워드로 6개월 이상 행동 변화 유지 가능.
- sMiles (App Store 현재 운영): Lightning 지갑 직접 출금 가능 sats 리워드. 핵심 리텐션 = "비트코인 직접 축적" 심리적 소유감.
- 피트니스 앱 평균 월 이탈률 9.2% — 스트릭/히스토리 없으면 초기 코호트 소멸 빠름.

### 운영 (R1)
- Blink 실계정 연결이 진짜 블로킹 조건. 테스트넷이 아닌 실계정 검증 일정 확보 필요.
- 지급 SOP: 누구나 실행 가능해야 함. SQL 콘솔 의존 불가.
- 관리 UI = 비트코인 지급 오류 방지 가드레일.

## R2 토론

### 기획 (R2)
- 안건 1 Phase A 확정 유지. 단 런칭 블로커가 아닌 "첫 지급 블로커"로 포지셔닝.
- 안건 2 Phase B 유지. 리서처의 "연속 운동 일수 카운터 Phase A 포함" 권고는 감독자 결정 사항으로 상향.
- 스코어 계산 기준: 개발의 `sum(points)` 단순화 지지 — 복합 스코어는 Phase B.

### 개발 (R2)
- UniqueConstraint 대상 재확인: `lightning_claims(user_id, week_label)` — reward_points는 동일 user/week에 다수 행이 정상.
- 악마의 변호인 "DB 직접 조회 대체" 반박: SOP 관점에서 SQL 콘솔 의존은 인수인계 불가 구조.
- 1일 공수는 read-only 집계 API + 어드민 UI 섹션 기준. 봇 차단 로직은 별도 의제.
- `_utc_today_start()` KST 불일치 버그 수용 — P1 수정 예정.

### 디자인 (R2)
- 4-탭 철회 입장 유지. Segmented Control on FeedPage 확정.
- Phase B `/history` 독립 페이지를 위해 Profile 탭 구현 시 `<ClaimHistoryList />` 컴포넌트 분리 권고 수용.

### QA (R2)
- UniqueConstraint P0 옵션: 옵션 A (Alembic 마이그레이션 + `CREATE UNIQUE INDEX`) 권장. 되돌리기 가능, 원자적.
- 신규 테스트 케이스 7종 정의 (아래 후속과제 참조).
- 악마의 변호인 "마이그레이션 Railway 재파괴 위험" 반박: 이번 변경은 인덱스 추가로 타입 변경과 리스크 클래스 다름.
- Blink 실계정 없으면 `/history` 주요 기능 검증 불가 → Phase B 결정 조건으로 Blink 실계정 연결 선행.

### 재무 (R2)
- 상위 10명 집중 지급 유지. 초기 소규모 지급 → 임팩트 높은 "당첨자" 마케팅 소재 생성.
- Blink 수수료 구조 확인 전까지 정확한 비용 계산 불가 — 감독자 확인 필요.

### 마케팅 (R2)
- 리더보드 공개 타이밍은 어뷰징 계정 1주 manual review 이후. QA와 정렬.
- "Real Bitcoin" 메시지 + 첫 지급 당첨자 스토리 콘텐츠 준비.

### 리서처 (R2)
- 악마의 변호인 "리더보드 바이럴 미검증" 반박: 조건부 참. 리더보드+공유 결합 시 피트니스 앱 K-factor 0.6~0.9 사례 복수 보고.
- Phase B History 캘린더 이연 리스크: 초기 코호트 17~20% 이탈 가능성 (9.2% 월간 이탈률 × 2개월 복리).
- **최소 리텐션 방어책**: Phase A에 "연속 운동 일수 카운터(숫자만)" 포함 강력 권고. 캘린더 없이도 스트릭 심리 구현 가능.

### 운영 (R2)
- Blink 실계정 = P0 블로커, 안건 1 = 첫 지급 블로커 (런칭 블로커 아님).
- 체크리스트:
  - Blink 실계정 연결 → 첫 지급 블로커
  - 안건 1 (관리 UI) → 첫 지급 블로커
  - 지급 SOP 문서화 → 첫 지급 블로커
- 악마의 변호인 "DB 직접 조회 대체" 반박: 비트코인 지급은 되돌리기 어려움. 관리 UI가 오류 경로 차단 가드레일.

### 악마의 변호인 (R2)
- ⚠️ 세션 한도 초과로 R2 미제출. R1 경고 잔존: "Blink 실계정 없는 리더보드는 신뢰를 깎을 수 있다."

   - 개발·QA·운영 동의. 악마의 변호인 R1 우려 → QA R2 반박으로 해소
   - `결정`

3. **안건 2 (History 캘린더 독립 페이지) = Phase B 이월**
   - 기획·개발·디자인 동의
   - Phase B 시작 조건: Blink 실계정 연결 확인
   - `결정`

4. **어드민 UI 설계: Segmented Control on FeedPage (이번 주 / 지난 주 토글)**
   - 4번째 탭 추가안 폐기. 기존 FeedPage 상단에 Segmented Control 삽입.
   - 디자인·개발·기획 동의
   - `결정`

5. **스코어 계산 기준 Phase A: `sum(points)` 단순 합계 고정**
   - 복합 스코어는 Phase B 독립 의제
   - QA: 테스트 명세 안정성 확보. 개발·기획 동의.

6. **Blink 실계정 연결 = 진짜 블로킹 조건**
   - 실계정 없으면 `/history` 기능 검증 불가, 지급 SOP 실행 불가
   - 기획·개발·운영·재무·QA 동의
   - `결정` (타이밍은 감독자 확인 필요)

### 미해결 쟁점

1. **리더보드 상위 N값**
   - QA, 개발 모두 미결 명시. top-10 vs top-50 vs 전체
   - LIMIT 값 미결정 → 테스트 기준 불확정

2. **본인 순위 별도 노출 여부**
   - QA, 디자인 스펙 미작성 상태
   - top-N 외 요청 사용자 본인 행 별도 포함 여부

3. **리워드 표시 단위**
   - sats 단가 표시 vs 원화 환산 병기
   - 기획·재무 미결

4. **Phase A "연속 운동 일수 카운터" 포함 여부**
   - 리서처 강력 권고 (이탈 방어 최소 조건)
   - 기획 Phase B 이월 입장 vs 리서처 Phase A 포함 권고
   - 충돌 미해소
