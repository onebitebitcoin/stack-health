## R1 의견

### 기획

- Phase A 목표는 2026-05-28 MVP 런칭과 3개월 내 사용자 100명 확보이므로, 공유 첫인상을 높이는 OG/title/description은 런칭 준비 범위에 부합한다.
- 단, 핵심 기능보다 우선하면 안 되며 전면 테마 개편은 보류해야 한다.
- 카피는 “운동 성장 커뮤니티 + 비트코인 후크” 구조가 비전에 맞다.
- 보상·BTC 표현은 법무 확인 전 보수적으로 운영해야 한다.

### 개발

- repo 근거: `frontend/index.html`은 `title`, `description`, Vite favicon 수준이고 OG/Twitter 메타가 없다.
- 로고는 `frontend/src/components/LogoMark.tsx` 인라인 SVG로 존재하며 `LoadingScreen.tsx`에서 이미 사용 중이다.
- 로그인 첫 화면은 `frontend/src/pages/LoginPage.tsx`에서 텍스트 중심이라 로고 노출 개선 여지가 있다.
- `frontend/src/tokens.css`, `frontend/tailwind.config.js`, `frontend/src/store/theme.ts`가 CSS 변수 기반 테마 구조를 제공하므로 소폭 컬러 조정은 가능하다.
- 정적 브랜드 공유 패키지는 낮은 공수이나, 개별 영상별 OG는 DB/API/서버 HTML 응답 설계가 필요해 Phase A에서는 비추천이다.

### 디자인

- OG/title/description은 공유·초대·SNS 유입에서 “앱이 완성돼 보이는가”를 결정하므로 Phase A 우선순위가 높다.
- 로고는 진입·공유·빈 상태·로딩에서 강하게, 콘텐츠 감상 중에는 작게 유지해야 한다.
- 추천 방향은 다크 차콜 + Volt 라임을 기본으로, 사토시 앰버는 보조 포인트로만 사용하는 것이다.
- 추천 description: “15초 운동 기록을 공유하고, 커뮤니티 반응과 스코어로 꾸준함을 이어가세요.”

### QA

- 검증 기준: `frontend/index.html`에 title/description/og:title/og:description/og:image/og:type/twitter:card가 존재해야 한다.
- OG image는 실제 배포 URL 기준 200 응답, MIME 타입, 크기/비율, 캐시 동작을 확인해야 한다.
- 로고 스타일 컬러 적용 시 `tokens.css`의 대비와 CTA/입력/BottomNav/어드민 가독성 회귀를 확인해야 한다.
- 최소 검증은 `cd frontend && npm run build`이며, 가능하면 기존 Playwright e2e를 함께 실행한다.

### 악마의 변호인

- 런칭 D-6 상황에서 브랜드 polish가 업로드/피드/리워드 안정성 검증을 밀어내면 실패한다.
- 현재 파일형 최종 로고/OG 자산이 확인되지 않으므로 임시 로고를 전체 브랜드로 과신하면 리브랜딩 비용이 커진다.
- 과하게 polished한 스플래시/OG가 실제 리워드 정책보다 앞서 보이면 스캠처럼 보일 수 있다.
- 정적 OG 하나만 넣고 “공유 최적화 완료”라고 착각하면 안 된다. 영상별 OG는 별도 설계가 필요하다.

### 재무

- 예상 비용은 낮고 지속 원가 증가는 거의 없다.
- 공유 링크 클릭률, 첫인상 신뢰도, 가입 전환율 개선 효과가 기대되므로 OG/title/description은 저비용 고효율 작업이다.
- 단, 전체 리디자인·고급 스플래시·외주 자산 구매는 초기 100명 검증 전 ROI가 낮다.

### 마케팅

- 초기 유입은 카카오톡, X, 인스타그램 DM/스토리, 커뮤니티 공유처럼 링크 미리보기가 첫 접점이다.
- 대표 OG 1장, title/description, favicon, 로그인/빈 상태 로고 노출은 P0 성격이다.
- “돈 버는 운동앱”, “비트코인 지급 보장”, “수익형 운동 플랫폼” 표현은 금지해야 한다.
- 추천 구성: 로고 + 운동 쇼츠/피드 느낌 + “운동 기록 · 커뮤니티 · 스코어” 3요소.

### 리서처

- 현재 안건은 신규 유입 첫인상 개선에 연결된다.
- 외부 최신 조사는 이번 R1에서 수행하지 않았으나 플랫폼별 OG 권장 규격, 캐시 정책, 규제성 표현은 별도 확인이 필요하다.
- 메시지는 “운동 기록을 공유하고 성장 피드백을 받는 커뮤니티 — 보상 경험 포함”이 안전하다.

### 운영

- Phase A에서는 전역 정적 메타데이터 1세트가 운영상 적합하다.
- 배포 전 체크리스트에 공유 미리보기 정상, BTC 과장 문구 없음, OG 이미지 경로 정상, favicon 반영, 로그인/빈 상태 로고 노출을 포함해야 한다.
- 동적 영상별 OG는 부적절 콘텐츠 노출, 삭제 후 캐시 잔존, 플랫폼별 크롤러 대응으로 운영 부담이 크다.

## R2 토론

### 기획 R2

다수 역할이 정적 OG/메타/로고 노출은 찬성했고, 악마의 변호인도 “범위 제한” 조건이면 핵심 리스크를 줄일 수 있다고 본다. Phase A 범위는 “런칭 공유 패키지”로 제한한다.

### 개발 R2

정적 OG와 favicon, 로그인/빈 상태 로고 노출은 기존 구조 안에서 가능하다. 반면 영상별 OG는 `Post`/`Video` 모델에 썸네일 필드가 없고 SPA fallback 구조라 별도 서버 렌더 설계가 필요하므로 Phase A에서 제외한다.

### 디자인 R2

로고를 좋아한다는 감독자 선호는 반영하되, 로고 과노출은 피한다. 콘텐츠 감상 화면보다 로그인, 로딩, 빈 상태, OG 이미지처럼 맥락상 적절한 공간에 노출한다. 기본 감성은 Volt Dark/차콜 + 라임, 앰버는 보조로 제한한다.

### QA R2

승인 조건은 명확하다. 메타 태그 존재, 정적 이미지 접근성, 빌드 성공, 주요 화면 가독성 회귀 없음이 확인되어야 한다. 캐시 확인은 배포 후 운영 체크리스트로 남긴다.

### 악마의 변호인 R2

전면 리디자인, 긴 스플래시, 보상 과장 문구, 동적 OG를 제외한다면 가장 큰 리스크는 완화된다. 다만 “정적 OG만으로 공유 최적화 완료”라는 착각과 “실제 리워드 신뢰보다 포장 먼저”라는 위험은 경고로 유지한다.

### 재무 R2

반나절~1일 수준의 정적 브랜드 패키지는 ROI가 좋다. 외주/유료 자산 구매나 대규모 리디자인은 Phase A 이후 데이터가 생긴 뒤 판단한다.

### 마케팅 R2

타이틀에 비트코인을 넣는 것은 후크로 유효하지만, 설명문은 운동 기록/커뮤니티/스코어를 중심에 둬야 한다. 과장 보상 문구는 사용하지 않는다.

### 리서처 R2

플랫폼별 권장 OG 규격과 캐시 갱신 방식은 구현 직전 최신 공식 문서/플랫폼 도구로 확인해야 한다. 법무성 표현은 “외부확인 필요”로 남긴다.

### 운영 R2

정적 OG는 SOP화가 쉽고 런칭 체크리스트에 넣을 수 있다. 변경 후 공유 미리보기 캐시 때문에 즉시 반영되지 않을 수 있음을 운영 리스크로 공지한다.

## 합의 및 결정

### 합의된 사항

1. **정적 대표 OG image/title/description/twitter card/favicon/app icon을 Phase A 런칭 전 추가한다. — 결정**
   - 동적 영상별 OG가 아니라 서비스 대표 공유 이미지 1세트로 시작한다.

2. **카피 방향은 운동 성장·커뮤니티·스코어 중심, 비트코인은 후크/보조로 제한한다. — 결정**
   - 금지 방향: “돈 버는 앱”, “지급 보장”, “수익형 운동 플랫폼”, “BTC 즉시 지급”처럼 확정 보상·수익을 암시하는 표현.

3. **로고는 적절한 공간에 추가 노출한다. — 결정**
   - 우선 위치: 로그인 화면, 로딩/스플래시성 화면, 빈 피드/빈 상태, OG 이미지.
   - 피드 콘텐츠 위 과한 워터마크는 하지 않는다.

4. **스플래시 페이지는 별도 대형 랜딩이 아니라 기존 첫 진입/로딩/로그인 경험 개선으로 제한한다. — 결정**
   - 목표는 “로고 + 한 줄 가치제안 + CTA” 수준의 첫인상 개선이다.

5. **테마 컬러는 로고 스타일에 맞춰 제한적으로 정리한다. — 결정**
   - 기본 방향: 다크 차콜/Volt 라임 중심, 사토시 앰버는 보조 포인트.
   - 기존 `tokens.css`와 Tailwind 변수 구조를 재사용한다.

6. **구현 후 최소 `frontend` 빌드를 실행하고, 가능하면 e2e/공유 미리보기 체크를 수행한다. — 결정**

### ⚠️ 악마의 변호인 미해소 경고

- **정적 OG만으로 공유 최적화가 끝난 것은 아니다.** 영상별 공유, 플랫폼 캐시, 부적절 콘텐츠 노출은 Phase B 설계 과제로 남는다.
- **브랜딩이 실제 리워드 신뢰를 대체할 수 없다.** OG/스플래시가 과하게 “돈 되는 앱”처럼 보이면 신뢰가 오히려 하락할 수 있다.
- **런칭 전 범위 확장을 금지해야 한다.** 전면 리디자인, 긴 스플래시, 동적 OG는 Phase A에서 제외한다.

- **P0 / 운영**: 배포 전 체크리스트에 공유 미리보기 정상, OG image 200 응답, BTC 과장 문구 없음, favicon 반영 확인을 추가한다.
- **P1 / 개발**: 로그인 화면과 빈 상태에 `LogoMark + wordmark`를 자연스럽게 노출한다.

- **P1 / 마케팅**: 최종 title/description/OG 문구 후보 2~3개를 정리해 감독자 결정으로 올린다.
- **P2 / 리서처**: 플랫폼별 OG 권장 규격과 캐시 갱신 도구를 최신 공식 문서 기준으로 확인한다.
- **P2 / 기획**: 영상별 동적 OG/썸네일 생성은 Phase B backlog로 분리한다.
