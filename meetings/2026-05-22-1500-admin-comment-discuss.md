## 현재 상태 (회의 전)

- `AdminPage.tsx`: 클레임 목록 + mark-paid + 테마 프리뷰만 존재
- `backend/app/routes/admin.py`: GET /admin/claims, PATCH /admin/claims/{id}/mark-paid
- `Video` 모델: id, user_id, r2_key, cdn_url, file_hash, duration_sec, status, created_at
- `Comment/Reply` 모델: 없음
- `VideoCard`: 좋아요·삭제 버튼만 있음, 댓글 UI 없음

## R1 의견

### 기획
- 어드민 3탭은 비트코인 리워드 지급 운영에 즉시 필요 — 없으면 수동 운영 불가
- 댓글은 Comment/Reply 모델 신규 + VideoCard UI + API 전체 부재 → 공수 과대
- 남은 런칭 6일 기준 동시 구현은 범위 초과
- **제안**: 어드민 3탭 P0, 댓글 Phase B 명시적 분리

### 개발
- 어드민 탭: `/admin/videos` 기존 존재, 기존 `require_admin` 패턴 재사용 → 1-2일 내 가능
- User 모델에 `is_banned` 컬럼 미존재 → 유저 밴은 DB 스키마 변경 + 마이그레이션 필요
- Comment/Reply 모델 전무 → 테이블 2개 신규 + API + React 컴포넌트 신규
- Reply(대댓글): parent_id self-referential FK → Comment보다 공수 2배
- **제안**: 어드민 탭 먼저, 댓글/대댓글은 Phase B

### 디자인
- 댓글 = 하단 드로어(bottom sheet) 패턴 — 영상 재생 유지, 댓글 레이어 분리
- VideoCard 우측 세로 버튼 스택: Heart → MessageCircle → Trash2 (Reels 스타일)
- Framer Motion `y` 애니메이션으로 슬라이드업, 영상 계속 재생
- Volt Dark 테마 bottom sheet: `bg-zinc-900/95 backdrop-blur`

### QA
- X-Admin-Key 단순 인증으로는 고위험 액션(ban, 리워드 조정, 영상 삭제) 불충분
- 댓글 DB 마이그레이션 + ban status 프로덕션 배포 중 롤백 전략 미확인
- **Action Log 테이블 필수** — 감사 추적 없으면 QA 불가
- 댓글은 MVP 분리, 도입 시 pytest E2E 최소 커버 후 머지

### 악마의 변호인
- 6일 내 신규 DB 테이블·API 6세트·UI 재구성 = 일정 파탄 교과서 패턴
- 어드민 없이도 현재 서비스 작동 중 → 6일 버틸 수 있다는 반증
- 댓글이 먼저 열리면 어뷰징 대응 수단 없이 유해 콘텐츠 채널을 먼저 뚫는 것
- **도전 질문**: "어드민/댓글 중 하나라도 2026-05-28 이후 추가해도 비즈니스 실패하는 근거가 있나?"

### 재무
- 댓글 데이터: R2 무료 10GB 티어 내 흡수 가능 (비용 영향 미미)
- 리워드 관리 탭: 수동 지급 오류율 감소 → 운영 비용 절감
- Railway 부하 모니터링 필요 (요청 2배 시 플랜 업그레이드 트리거)

### 마케팅
- 댓글이 기록→반응→기록 루프를 완성하는 핵심 레이어
- 초기 100명 미만 환경에서 빈 댓글창 = "아무도 없는 앱" 이탈 신호
- **제안**: 어드민에서 운영자 시드 댓글 입력 기능 포함

### 리서처
- Strava 데이터: 그룹 활동 시 소셜 반응 2배 → 리텐션 직결
- 댓글 배포 전 어드민 신고·숨김 기능 선행 필요
- **제안**: 신고 기능 완성 후 팔로워 전용 → 전체 공개 단계적 오픈

### 운영
- 현재 영상 삭제·유저 정지 어드민 수단 없음 → 운영 공백
- 소규모 팀에서 어드민 + 댓글 모더레이션 동시 처리 시 리워드 SOP 집중력 분산
- **제안**: 어드민 탭 먼저, 댓글은 신고 자동 플래그(임계값 N회 신고 시 자동 숨김) 포함 2단계

## R2 토론

### 기획 R2
- 동의: 운영 2단계 릴리스, 리서처 신고·숨김 선행
- **반박**: 마케팅 시드 댓글 = 범위 팽창, Phase B 후순위
- **최종**: 어드민 3탭 5.28 P0, 댓글은 신고·숨김 포함 Phase B로 종결

### 개발 R2
- 동의: QA Action Log 필수(현재 감사 추적 미구현), 운영 2단계 순서
- **반박**: 마케팅 시드 댓글 = 댓글 테이블 없이 불가, Phase B 완료 후 논의
- **최종**: 어드민 3탭 1일 내 가능, 댓글은 Comment 모델+Alembic+API+UI 별도 Phase

### 디자인 R2
- 동의: 마케팅 "빈 댓글창 이탈" 우려(UX 타당), QA Action Log
- **반박**: 리서처 "신고·숨김 선행" → 어드민 수동 삭제로 대체 가능, 영상 탭 상태 뱃지 컬럼이 더 빠름
- **최종**: 어드민 3탭 현 스프린트, 댓글 UX 설계(bottom sheet+버튼 스택) 확정, 구현은 Phase B

### QA R2
- 동의: 운영+악마 "댓글 오버스코프", 마케팅 시드 댓글 댓글 테이블 없이 불가
- **반박**: QA 제안 → 댓글 DB 스키마만 이번 스프린트에 설계, 기능 구현은 Phase B
- **최종**: Action Log + 어드민 인증 강화 P0, 댓글 스키마 설계만 포함 후 기능은 Phase B

### 악마의 변호인 R2
- 도전 질문 응답 평가: 부분적 — "즉시 필요"는 당위론, 실제 인과관계 미증명
- **미해소 리스크**: 공백이 있어도 6일 버텨온 게 이미 반증. 댓글 = 어뷰징 채널 최초 개방 위험
- **최종 경고**: 어드민보다 댓글이 먼저 열리면 순서 역전 — 어드민 먼저가 유일한 올바른 순서

### 마케팅 R2
- 동의: 리서처 신고 선행, 운영 2단계
- **반박**: 기획 "댓글 Phase B 완전 분리" → 시드 댓글 1개 기능만 Phase A 어드민에 포함하면 이탈 완화 가능
- **최종**: 어드민 3탭 즉시, 시드 댓글 어드민 선포함 후 공개 댓글 단계적 오픈

### 운영 R2
- 동의: 악마 "댓글 P0 아님", QA Action Log 필수
- **반박**: 마케팅 시드 댓글 = SOP(작성 기준·승인·삭제 규정) 없으면 모더레이션 리스크
- **최종**: 어드민 3탭 이번 주, 댓글은 신고 SOP 완성 후 다음 스프린트

## 합의 및 결정

### 합의된 사항

1. **결정**: 어드민 3탭(유저 관리·영상 콘텐츠 관리·리워드 관리)은 Phase A P0로 2026-05-28 런칭 전 완성한다.
   - 지지: 기획, 개발, 디자인, QA, 운영, 악마 (6개 역할)
   - 개발 공수: 기존 `require_admin` 패턴 재사용, 1일 내 가능

2. **결정**: 댓글(Comment) 기능은 Phase B로 명시적 분리한다. 어드민 신고·숨김 기능이 먼저 완성된 후 댓글을 공개한다.
   - 지지: 기획, 개발, QA, 운영, 악마 (5개 역할)

3. **결정**: 어드민에 Action Log 테이블을 추가한다 (ban, 영상 삭제, 리워드 조정 감사 추적).
   - 지지: QA, 개발, 운영 (3개 역할)

4. **결정**: 댓글 UX 설계를 이번 스프린트에 확정한다 (구현은 Phase B):
   - VideoCard 우측 세로 버튼 스택: Heart → MessageCircle → Trash2
   - 댓글 = bottom sheet (영상 계속 재생, `bg-zinc-900/95 backdrop-blur`)
   - Framer Motion `y` 슬라이드업 애니메이션
   - 지지: 디자인, 기획, 마케팅 (3개 역할)

5. **결정**: 어드민 탭별 구체 범위:

   | 탭 | 기능 |
   |---|---|
   | **유저 관리** | 유저 목록, 정지(ban) 토글, 가입일·영상수 표시 |
   | **영상 콘텐츠** | 영상 목록, 강제 삭제, 상태 뱃지(활성/숨김/신고) |
   | **리워드 관리** | 클레임 목록 (기존), mark-paid, 스코어 현황 |

### 미해결 쟁점

1. **미해결**: 어드민 시드 댓글 기능 Phase A 포함 여부
   - 마케팅: 빈 댓글창 이탈 방지에 필요, 1개 기능이면 충분
   - 기획·개발·운영: 댓글 테이블 없이 불가, 범위 팽창, SOP 부재 리스크

2. **미해결**: 댓글 DB 스키마를 이번 스프린트에 설계만 할지
   - QA: 스키마 설계만 포함 제안
   - 기획: 완전 분리 주장
