### C-1. [CRITICAL] 분배 지급이 비원자적 + 재진입/멱등 가드 없음 → 실 sats 이중 지급
- **위치**: `backend/app/services/mining.py:128-153` (지급), `mining.py:69-75`(재선택), `mining.py:112-123`(라운드 상태)
- **근거**:
  ```python
  for user_id, sats_won in winnings.items():
      claim = claim_by_uid.get(user_id)
      ...
      if do_pay and settings.blink_api_key:
          result = blink_service.pay_lightning_address(claim.ln_address, sats_won)  # 실 sats 전송(되돌릴 수 없음)
          if result["success"]:
              claim.status = "paid"      # 메모리에만 반영
          ...
  db.flush()                              # 루프 후 단 1회 flush, commit은 라우트(admin.py)에서
  ```
  - run_lottery는 `status=="pending"` claim을 선택(L73). 지급 루프 중간에 commit이 없고, 최종 `db.flush()`(L153) 후 commit은 호출자(admin.py run-lottery 라우트)에서 일괄 수행.
  - `mining_round.status`를 실행 전 검사하지 않음(L112-123은 생성/갱신만). **재진입 가드 부재**.
- **영향(트리거+파급)**:
  1. **부분 지급 후 트랜잭션 롤백**: Blink로 A·B에게 실제 sats 전송 성공 → 이후 `db.flush()`/라우트 commit이 실패(DB 커넥션 끊김, 프로세스 크래시, 예외)하면 A·B의 `status="paid"`가 **커밋되지 않고 롤백**. DB는 여전히 `pending`. 같은 week로 재실행 시 A·B가 다시 선택되어 **재지급**.
  2. **동시 실행/더블클릭**: admin이 run-lottery를 두 번 호출하면 두 실행이 같은 `pending` claim 집합을 선택(L69) → **동시 이중 지급**.
  - 멱등 키·지급 원장·`mining_round.status=="distributed"` 사전 검사 중 어느 것도 없음. 실제 비트코인 손실로 직결되는 최상위 리스크.
- **수정**:
  - 실행 시작 시 `mining_round.status`가 이미 `distributed`면 거부(재진입 차단).
  - 지급 직전 claim을 원자적으로 `paid`로 선점(예: `UPDATE ... WHERE status='pending' RETURNING` 또는 행 잠금) 후 Blink 호출, 건별 commit. 멱등 키(claim_id 기준)로 Blink 중복 호출 방지.
  - Blink 성공 ↔ DB 상태를 건별 트랜잭션으로 묶고, 실패 건은 별도 재시도 큐로.

### C-4. [LOW] points_to_sats float→int 절사로 최대 1 sat 손실
